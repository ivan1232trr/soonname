# CityPulse — AI-Generated Code Policy

**Version:** 1.1
**Status:** Active
**Date:** March 2026
**Applies to:** All code in this repository

---

## 1. Policy Statement

Every file or code block generated by an AI system (including Claude, GitHub Copilot, ChatGPT, or any other AI assistant) must be explicitly labeled at the point of creation and tracked in this document. This policy applies from day one and covers the entire codebase.

The expectation is that the majority of code in this repository will be AI-generated. That is acceptable. What is not acceptable is AI-generated code that is unattributed, unreviewed, unaccounted for, or unexplained.

**Every single line of generated code must be commented.** No line is too obvious to comment. If a line queries the database, the comment says what it is querying and why. If a line filters an array, the comment explains what is being filtered and what the result is used for. A reader must be able to understand the intent of every line without needing to trace execution elsewhere.

---

## 2. Why This Policy Exists

- **Accountability** — Every line of code must have an identifiable origin. "The AI wrote it" is not an excuse for not understanding it.
- **Comprehension** — A comment on every line forces the author to understand what the line does before shipping it. AI can generate plausible-looking code that does the wrong thing silently.
- **Review standards** — AI-generated code requires human review before merge. Line-level comments make that review faster and more thorough.
- **Debugging clarity** — When something breaks, commented code tells you what the author believed the line was doing — which is exactly what you need to find where the belief was wrong.
- **Onboarding** — New team members can read the codebase without needing to reverse-engineer intent from bare logic.
- **Intellectual property** — Labeling AI-generated code creates a clear record for any future IP or licensing questions.
- **Trust** — A codebase where AI involvement is transparent and every decision is explained is more trustworthy than one where it is hidden.

## 3. Labeling Convention

### 3.1 File-Level Label

If an entire file was generated by AI, include the following comment block at the top of the file, immediately after any shebang line or license header:

**JavaScript / TypeScript:**

```js
// Tool: Claude (claude-sonnet-4-6)
// Date: YYYY-MM-DD
// Prompt summary: [one-line description of what was asked for]
// Reviewed by: [name or "unreviewed"]
```

If only a portion of a file was AI-generated (e.g. a single function inserted into a human-written file), wrap that block:

**JavaScript / TypeScript:**
```js

  // ...
```

**Python:**

## 4. Line-by-Line Comment Standard

This is a hard rule. **Every line of generated code must carry a comment explaining what it does and why.**

This is not about adding a comment to every function. It is about commenting every individual statement — every declaration, every query, every condition, every return.

### 4.1 What a Comment Must Say

A valid comment answers at least one of:
- **What** — what this line produces or does
- **Why** — why this operation is needed here
- **Where the result goes** — what the calling code will do with it

A comment that simply restates the syntax in English (e.g. `// set x to 5`) is not sufficient if it does not explain the purpose.

### 4.2 Examples

**Database query:**
```ts
// Fetch all upcoming events within 50km of the user's city center, ordered by match score descending
const events = await db.events.findMany({
  where: {
    cityId: user.cityId,           // restrict results to the user's selected city
    date: { gte: new Date() },     // exclude events that have already passed
  },
  orderBy: { matchScore: 'desc' }, // highest AI-ranked events appear first in the feed
});
```

**Array filter:**
```ts
// Remove events the user has already marked as Interested — they appear in the Saved tab, not the feed
const unseenEvents = events.filter(event => !user.interestedIds.includes(event.id));
```

**Conditional:**
```ts
// If the AI classifier did not assign a category, default to 'Entertainment' so the event is always visible on the map
const category = event.category ?? 'Entertainment';
```

**Score calculation:**
```ts
// Compute a weighted overlap: count how many of the event's tags appear in the user's interest profile
const tagOverlapScore = event.tags.filter(tag => userProfile.tags.includes(tag)).length;

// Apply a distance penalty — events more than 10km away score proportionally lower
const distancePenalty = Math.max(0, (distanceKm - 10) * 0.05);

// Final match score combines tag relevance and distance; higher is better
const matchScore = tagOverlapScore - distancePenalty;
```

**Return statement:**
```ts
// Return the top 50 ranked events; the feed paginates beyond this on scroll
return rankedEvents.slice(0, 50);
```

### 4.3 Comment Placement

- **Inline comments** on the same line are acceptable for short, self-contained lines.
- **Line-above comments** are preferred when the explanation needs more than ~60 characters.
- For multi-step operations (e.g. a query with multiple `where` clauses), comment the block header and each clause individually, as shown in the examples above.

### 4.4 What Fails This Standard

The following are not acceptable:

```ts
// bad — restates the syntax, explains nothing
const events = await db.events.findMany({ where: { cityId: user.cityId } }); // find events

// bad — vague
const score = tagOverlap - penalty; // calculate score

// bad — no comment at all
return events.slice(0, 50);
```

Every function and class must also have a block comment above it that describes:
1. What it does (one sentence)

3. What it returns (type and meaning)
4. Any side effects (database writes, external API calls, mutations)

## 6. Review Requirements

AI-generated code is not automatically production-ready. The following rules apply before AI-generated code can be merged:

| Code Type | Review Requirement |
|---|---|
| Business logic, AI integration, auth | Mandatory human review. Must be understood by the reviewer, not just approved. |
| UI components, styling | Spot-check review. Confirm behavior matches UI requirements. |
| Configuration files, scripts | Full review. Misconfigured tooling causes outsized damage. |
| Tests | Full review. AI-generated tests that test nothing are worse than no tests. |
| Documentation / markdown | Light review. Confirm accuracy against actual implementation. |

**Rule:** The `Reviewed by` field in the file header and in the registry must be filled in before any file is merged to the main branch. "Unreviewed" is a valid temporary state for a work-in-progress branch, but not for a merge.

## 7. What Does Not Need Labeling

The following do not require AI labeling:

- Content copied from official documentation (README examples, etc.)
- Boilerplate from official CLI tools (`create-next-app`, `create-react-app`, etc.) — these should be noted in commit messages instead
- Minor AI-assisted autocomplete on individual tokens (single variable names, etc.) — only label when a meaningful block of logic was generated

Use judgment. If an AI generated the structure, the logic, or the approach — label it. If AI autocompleted a closing bracket — skip it.

## 8. Enforcement

- Pull request template (to be added) will include a checklist item: "All AI-generated code in this PR is labeled, commented line by line, and added to the registry."
- Code review must reject PRs where AI-generated code is unlabeled or contains uncommented lines.
- The registry in this file is the canonical record. Keep it updated.

## 9. Document + Non-Code Files

This policy covers code files. For non-code AI-generated content:

| Asset Type | Labeling Approach |
|---|---|
| Planning documents (PRD, UI requirements) | Footer note: *"Prepared with AI assistance (Claude, YYYY-MM-DD)"* |
| Copy / marketing text | Note in the document or in a companion notes file |
| Design files | Layer/frame naming convention in the design tool (e.g. prefix with `[AI]`) |
| Database migration scripts | Treated as code — full labeling and line commenting required |
