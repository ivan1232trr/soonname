# CityPulse — Data Requirements Document

**Version:** 1.0
**Status:** Draft
**Date:** March 2026
**Companion to:** UI Requirements v1.0, Technical Decisions v1.3
**Audience:** Database engineer, backend engineer, anyone responsible for schema design

---

## Purpose

This document defines what data CityPulse collects, who or what produces it, and how each field must be stored. It is the authoritative reference for database schema design. Every table, column, constraint, and index decision should be traceable to a requirement in this document.

This document does not specify how to write Prisma schema syntax or SQL — it specifies what must exist and why. Implementation rules for PostgreSQL + Prisma are in TD-004.

---

## Data Sources

Before listing entities, it helps to understand where data comes from. There are four sources in CityPulse:

| Source | Description |
|---|---|
| **User input** | Data the user explicitly types or selects — registration, onboarding, event submissions |
| **Behavioral signal** | Implicit data generated by user actions — tapping an event card, saving, sharing, skipping |
| **AI system** | Data produced by the Claude classifier — category assignments, tag assignments |
| **Backend computation** | Data derived by the server from other data — H3 indexes computed from lat/lng |

## Entities

### 1. User

**What it is:** A registered account. The minimum identity record needed to submit events, save preferences, and accumulate interaction history.

**Who provides it:** The user, at registration.

**When it is created:** On account creation (sign-up flow — not defined in MVP UI scope but required for any saved data to exist).

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | Generated at record creation; never exposed as a sequential integer |
| `email` | String | User input | Yes | Unique, max 255 chars, valid email format | Used for authentication; not displayed in the UI |
| `name` | String | User input | Yes | Max 100 chars | Display name shown on event submissions ("Posted by") |
| `passwordHash` | String | Backend (derived) | Yes | Not null | bcrypt or argon2 hash of the user's password; raw password is never stored |
| `createdAt` | Timestamp | Backend (generated) | Yes | Default: now(), UTC | When the account was created |
| `updatedAt` | Timestamp | Backend (generated) | Yes | Auto-updated on any change, UTC | When the account was last modified |

**Indexes required:** Unique index on `email` (login lookup).

**Relationships:**
- One User → one UserProfile (1:1)
- One User → many Events (submitted events)
- One User → many UserInteractions

### 2. UserProfile

**What it is:** The user's preference and interest data. This is the personalisation layer — it stores the onboarding answers and evolves as the user interacts with the app.

**Who provides it:**
- Onboarding (Steps 2–4): vibe preference, time preferences, initial interest tags — explicitly provided by the user
- Profile edit screen: user can update all fields post-onboarding
- Reset: all behavioral data cleared; profile reverts to last explicitly-set state

**When it is created:** Immediately after the user completes onboarding (before they see the feed for the first time). It is populated with the onboarding answers.

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | |
| `userId` | UUID | Backend (FK) | Yes | FK → User.id, unique (one profile per user) | |
| `preferredCityId` | UUID | User input (onboarding Step 1) | Yes | FK → City.id | City selected in onboarding; updated if user changes city in profile |
| `vibe` | Enum | User input (onboarding Step 2) | Yes | One of: `CHILL`, `SOCIAL`, `BOTH` | Directly maps to the three options on the Vibe screen |
| `timePreferences` | String[] | User input (onboarding Step 3) | Yes | Array; each value one of: `MORNING`, `AFTERNOON`, `EVENING`, `LATE_NIGHT`; min 1 item | Multi-select from the Time Preference screen |
| `interestTags` | String[] | User input (onboarding Step 4) | Yes | Array of tag slugs; min 3 items | Tags selected on the Interest Tags screen; slugs must exist in the Tag table |
| `updatedAt` | Timestamp | Backend (generated) | Yes | Auto-updated, UTC | When any preference was last changed |

**Indexes required:** Unique index on `userId`.

**Relationships:**
- One UserProfile → one User (1:1, inverse)
- UserProfile.preferredCityId → City.id
- UserProfile.interestTags are slug references into the Tag table (not FK-enforced — stored as a string array for flexibility, validated at the application layer)

**Notes:**
- `interestTags` stores slugs as a string array rather than a join table. The AI ranking engine reads this array directly to compute tag overlap scores. A join table would be structurally cleaner but adds join overhead on every feed request.
- The Profile screen shows a "Reset Interest Profile" action. When triggered, `interestTags`, `vibe`, and `timePreferences` are cleared and the user is returned to the onboarding flow. This is a full field reset, not a soft delete.

### 3. City

**What it is:** A supported city in CityPulse. Cities are the primary scope boundary — all events, feeds, and map views are scoped to one city at a time.

**Who provides it:** Backend / seed data. Cities are not user-created in MVP. The first record is Kingston, Jamaica (TD-011).

**When it is created:** Seeded at database initialisation. New cities are added via a data migration, not through the user-facing app.

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | |
| `name` | String | Seed data | Yes | Max 100 chars, unique | Human-readable city name (e.g. "Kingston") |
| `country` | String | Seed data | Yes | Max 100 chars | Country name or ISO code (e.g. "Jamaica") |
| `latitude` | Float | Seed data | Yes | WGS-84, range -90 to 90 | City center latitude; default map origin |
| `longitude` | Float | Seed data | Yes | WGS-84, range -180 to 180 | City center longitude; default map origin |
| `timezone` | String | Seed data | Yes | IANA timezone string, max 50 chars | Used to render event times in local time. Kingston = `"America/Jamaica"` (UTC-5, no DST) |
| `createdAt` | Timestamp | Backend (generated) | Yes | Default: now(), UTC | |

**Indexes required:** Unique index on `name`.

**Relationships:**
- One City → many Events
- One City → many UserProfiles (via preferredCityId)

### 4. Tag

**What it is:** The taxonomy of descriptive labels used to classify events and match them to user interests. Tags are the shared vocabulary between the event classifier and the user profile.

**Who provides it:** Defined and maintained by the CityPulse team. The initial tag set is seeded at database initialisation. Tags may be added via data migration as the classifier evolves — never user-created.

**When it is created:** Seeded at database initialisation. Updated only via deliberate schema migrations.

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | |
| `name` | String | Seed data | Yes | Lowercase slug format (e.g. `"live-music"`, `"family-friendly"`), max 50 chars, unique | The canonical label; used in UI chip display and as the reference value in UserProfile.interestTags |

**Seed tag examples (not exhaustive):**
`music`, `art`, `food-and-drink`, `fitness`, `nightlife`, `outdoors`, `tech`, `comedy`, `film`, `wellness`, `markets`, `sports`, `free`, `community`, `18-plus`, `outdoor`, `live-music`, `family-friendly`

**Indexes required:** Unique index on `name`.

**Relationships:**
- Tags are associated with Events via a join table (EventTag — see below).
- Tags are referenced by slug string in UserProfile.interestTags.

### 5. Event

**What it is:** The core content record. An event submitted by a user, visible on the feed and map once classified by the AI.

**Who provides it:**
- Core fields (title, description, location, date/time): **user input** via the Event Submission screen
- Category: **user input** (optional hint) or **AI system** (assigned after classification)
- Tags: **AI system** (assigned after classification); user may edit in the AI Preview step before submitting
- H3 indexes: **backend computation** — derived from lat/lng at write time, never supplied by the client
- Status: **backend** — set to `PENDING_CLASSIFICATION` on submission; updated to `ACTIVE` by the AI worker on completion

**When it is created:** On successful form submission (POST /events). Immediately visible in the feed as a "Pending classification" card.

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | |
| `title` | String | User input | Yes | Max 100 chars | Character count enforced on the submission form |
| `description` | String | User input | Yes | Min 20 chars, no max enforced | Primary input for the AI classifier |
| `locationName` | String | User input | Yes | Max 200 chars | Venue or area name as typed — freeform, not geocoded to a canonical address |
| `latitude` | Float | User input (via location picker or manual entry) | Yes | WGS-84, range -90 to 90 | Must be provided; geocoding step may be needed if user enters freeform text |
| `longitude` | Float | User input (via location picker or manual entry) | Yes | WGS-84, range -180 to 180 | |
| `h3R7` | String | Backend computation | Yes | 15-char hex string, not null | Computed from lat/lng at resolution 7 (~5 km²). Indexed — used for map clustering |
| `h3R9` | String | Backend computation | Yes | 15-char hex string, not null | Computed from lat/lng at resolution 9 (~0.1 km²). Indexed — used for proximity feed queries |
| `h3R11` | String | Backend computation | Yes | 15-char hex string, not null | Computed from lat/lng at resolution 11 (~895 m²). Display only — no index required |
| `category` | Enum | AI system (or user hint) | Yes on ACTIVE events | One of: `NIGHTLIFE`, `SPORTS`, `EDUCATION`, `FOOD`, `WELLNESS`, `CULTURE`, `ENTERTAINMENT`; nullable until classification completes | Nullable at submission time; AI worker fills this field. If the user provided a category hint, it is passed to the classifier but the AI result is authoritative |
| `eventDate` | Date | User input | Yes | ISO date `YYYY-MM-DD`; must be in the future at submission time | Calendar day the event occurs |
| `startTime` | Timestamp | User input | Yes | UTC-stored ISO datetime; rendered in city timezone in the UI | Start time |
| `endTime` | Timestamp | User input | No | UTC-stored ISO datetime; nullable | End time; omit if open-ended |
| `status` | Enum | Backend | Yes | One of: `PENDING_CLASSIFICATION`, `ACTIVE`, `FLAGGED`; default: `PENDING_CLASSIFICATION` | Controls feed visibility. Only `ACTIVE` events appear in the public feed |
| `cityId` | UUID | User input / session | Yes | FK → City.id | Scopes the event to a city; derived from the user's selected city at submission time |
| `submittedById` | UUID | Session (authenticated user) | Yes | FK → User.id | Not taken from the form — resolved from the authenticated session |
| `createdAt` | Timestamp | Backend (generated) | Yes | Default: now(), UTC | |
| `updatedAt` | Timestamp | Backend (generated) | Yes | Auto-updated on change, UTC | |

**Indexes required:**
- B-tree index on `h3R9` (proximity feed queries — primary spatial lookup)
- B-tree index on `h3R7` (map clustering queries)
- Index on `cityId` (all feed queries are city-scoped)
- Index on `status` (feed only shows ACTIVE events; combined with cityId in practice)
- Index on `eventDate` (filtering future events)

**Relationships:**
- One Event → one City (many events per city)
- One Event → one User (submitter)
- One Event → many EventTags (join table, see below)
- One Event → many UserInteractions

### 6. EventTag (Join Table)

**What it is:** The many-to-many relationship between Events and Tags. A single event can have many tags; a single tag can belong to many events.

**Who provides it:** The AI classifier populates this after classification. The user may pre-populate it via the AI Preview step on the submission form.

**When it is created:** Records are written by the AI worker when it finishes classifying an event. If the user edited tags in the AI Preview step, those edits are the starting state passed to the worker.

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `eventId` | UUID | Backend (FK) | Yes | FK → Event.id | |
| `tagId` | UUID | Backend (FK) | Yes | FK → Tag.id | |

**Primary key:** Composite `(eventId, tagId)` — no separate `id` column needed.

**Indexes required:** Index on `tagId` to support reverse lookups (find all events with a given tag).

### 7. UserInteraction

**What it is:** A log of every meaningful action a user takes on an event. This table feeds the AI ranking engine's personalization layer — the more interactions recorded, the better the feed ranking becomes.

**Who provides it:** Behavioral signals from the user. These are generated implicitly by the app as the user navigates — the user does not "fill in" an interaction; they just use the app.

**When it is created:** Immediately when the user performs the action (client fires an API call on every recorded interaction).

| Field | Type | Source | Required | Constraints | Notes |
|---|---|---|---|---|---|
| `id` | UUID | Backend (generated) | Yes | Primary key, unique | |
| `userId` | UUID | Session | Yes | FK → User.id | Resolved from the authenticated session |
| `eventId` | UUID | Client | Yes | FK → Event.id | Which event the user interacted with |
| `type` | Enum | Client | Yes | One of: `VIEW`, `SAVE`, `FLAG`, `SHARE` | See interaction type definitions below |
| `createdAt` | Timestamp | Backend (generated) | Yes | Default: now(), UTC | Interaction timestamp; used for recency weighting in ranking |

**Interaction type definitions:**

| Type | Trigger | Ranking signal |
|---|---|---|
| `VIEW` | User opens the Event Detail screen | Weak positive signal |
| `SAVE` | User taps the "Interested" button (toggleable) | Strong positive signal |
| `FLAG` | User reports the event (future — reserved in MVP) | Moderation signal; no ranking weight |
| `SHARE` | User shares the event externally | Moderate positive signal |

**Note on SAVE toggle:** The "Interested" button is toggleable. A `SAVE` interaction is written on tap. If the user un-saves, a second `SAVE` record is not written — instead, the existing record is deleted. The net presence or absence of a SAVE record for a (user, event) pair is the source of truth.

**Indexes required:**
- Composite index on `(userId, eventId)` — checking if a user has saved a specific event
- Index on `userId` — fetching all interactions for a user (profile ranking computation)
- Index on `eventId` — fetching all interactions on an event (moderation, popularity signals)

## Entity Relationship Summary

```
City
 └── has many Events
 └── referenced by many UserProfiles (preferredCityId)

User
 └── has one UserProfile
 └── has many Events (submitted)
 └── has many UserInteractions

UserProfile
 └── belongs to one User
 └── references one City (preferredCityId)

Event
 └── belongs to one City
 └── belongs to one User (submitter)
 └── has many EventTags → Tags (via join table)
 └── has many UserInteractions

Tag
 └── has many EventTags → Events (via join table)
 └── referenced by slug string in UserProfile.interestTags

UserInteraction
 └── belongs to one User
 └── belongs to one Event

EventTag (join)
 └── links Event ↔ Tag
```

## Data the System Does NOT Store

These are things the UI implies but that are intentionally excluded from the data model.

| Data | Reason Not Stored |
|---|---|
| Raw event images / media | Event images are out of scope for MVP (UI Requirements §12). Object storage is provisioned (TD-006) but not actively written to until a feature requires it. |
| AI match score / ranking score | Per TD-007 / UI Requirements §5.3, the ranking score is never shown to the user and is not persisted. It is computed per-request in the feed ranking engine. |
| User location history | The user's current position is computed at query time from the device and is never stored in the database (TD-001, R5). |
| Raw behavioral weights | The AI tag weights derived from interaction history are computed at ranking time, not stored as a separate field. `UserProfile.interestTags` is the only persisted profile signal in MVP. |
| Session tokens (in PostgreSQL) | Sessions are stored in Redis (TD-005), not in the relational database. They are not in scope for this document. |
| Password reset tokens | Token-based flows (forgot password, email verification) are not yet scoped. When added, they will likely be stored in Redis with a TTL, not in PostgreSQL. |

## Privacy and Data Lifecycle

### User-facing privacy controls (from UI Requirements §4.6.3)

| Control | Data Effect |
|---|---|
| **View my data** | No data change — read-only summary of what is stored |
| **Delete my data** | All records for this user are deleted: User, UserProfile, UserInteractions. Events submitted by the user remain but `submittedById` is set to null (anonymised) rather than cascade-deleted, to preserve event data integrity. |
| **Reset interest profile** | Clears `UserProfile.interestTags`, `vibe`, and `timePreferences`. Deletes all `UserInteraction` records for this user. The User and UserProfile records themselves are retained. |

### Retention

No automated retention policy is defined for MVP. All records persist until deleted by the user or by a manual admin operation. A formal retention policy must be defined before a public launch.

### PII fields

| Table | PII Fields | Notes |
|---|---|---|
| User | `email`, `name` | Email is used for authentication and is never shown publicly. Name is displayed publicly on event submissions. |
| UserProfile | `preferredCityId`, `interestTags`, `vibe`, `timePreferences` | Preference data; personal but not directly identifying on its own |
| UserInteraction | All fields (the record itself ties a user to an event) | Behavioral data; treat as personal data under GDPR |

## Timezone Handling

All timestamps in the database are stored in **UTC**. Conversion to local time (Jamaica Standard Time, UTC-5, no DST) happens in the application layer at display time, using the `City.timezone` field as the IANA reference.

The `eventDate` field (ISO date `YYYY-MM-DD`) is a calendar date without timezone. It represents the local date of the event, as entered by the user. It is not converted.

## Open Questions

These questions must be resolved before the schema is finalised.

| # | Question | Impact |
|---|---|---|
| 1 | Should `UserProfile.interestTags` be a string array or a proper join table (UserProfileTag)? | Array is simpler and faster for ranking reads; join table is more relational and enables tag-level analytics. Decision affects how profile resets and tag deletions are implemented. |
| 2 | How is user authentication handled? Is there a separate `AuthCredential` table for passwords and OAuth tokens, or does `passwordHash` live directly on `User`? | Affects whether social login (Google, Apple) can be added later without a schema change. |
| 3 | What happens to a `UserInteraction` of type `SAVE` when the user toggles "Interested" off — delete the row or add a second row with a `removed` flag? | Delete-on-toggle is simpler but loses history; flag approach preserves a complete interaction log for future analytics. |
| 4 | Should `Event.category` be nullable until classification completes, or should a default value (e.g. `UNCATEGORIZED`) be used? | Nullable is more semantically precise but requires null checks everywhere the field is read. An `UNCATEGORIZED` enum value avoids nullability at the cost of a less meaningful default. |
| 5 | Does the City table need an H3 boundary cell set column (a compact set of H3 cells representing the city boundary)? | TD-001 mentions this as a "city-scoping query" mechanism. Needed if city boundaries are used to restrict event visibility or feed scope. |
| 6 | What are the spam / rate-limit rules for event submission? | Not a schema question per se, but affects whether any per-user submission count or cooldown fields are needed on the User or a dedicated RateLimit table. |
